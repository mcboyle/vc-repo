🔒 External-Claims Verification: Hardened VeraCrypt 1.26.29 Fork — Source & Vendor/Patent Pass

**TL;DR**
- The single actionable finding is CONFIRMED and load-bearing: BIP-173 bech32 has the documented "trailing-p / inserted-q" insertion-deletion weakness that motivated BIP-350 bech32m (final XOR changed from 1 to 0x2bc830a3), and bech32's 4-substitution detection guarantee only holds up to 89 characters (90-char total cap) — so a ~118-character hand-transcribed share code using bech32 (^1) loses the guarantee entirely. Move recovery-share encoding to bech32m or, better, BIP-93 codex32.
- The XCB material is essentially correct and decisive: XCB-AES (IEEE 1619.2) is broken (two independent CRYPTO 2025 papers), SISWG has moved to remove it, and the report date conflict resolves in R25's favour (2 September 2024, not 9 February). The apparent "one ePrint, two papers" conflict is real but benign: ePrint 2024/1554 is a single evolving paper.
- The EME2 "phantom patent" correction is substantially CONFIRMED (only filing was the Regents-of-California US 2004/0131182, Rogaway inventor, abandoned) but R25's "UC San Diego" is wrong (Rogaway is UC Davis; assignee is the Regents of the University of California). **Details**

**PRIORITY 1 — BIP-350 / bech32m (CONFIRMED)**
The specific technical claim is verbatim correct. BIP-350 states: "Bech32 has an unexpected weakness: whenever the final character is a 'p', inserting or deleting any number of 'q' characters immediately preceding it does not invalidate the checksum." BIP-350 fixes this by "replacing the constant 1 that is xored into the checksum at the end with 0x2bc830a3." (Sources: bips.dev/350, github.com/bitcoin/bips bip-0350.mediawiki, en.bitcoin.it/wiki/BIP_0350.)
- (a) The 4-error guarantee and length: BIP-173 implements a BCH code that "guarantees detection of any error affecting at most 4 characters and has less than a 1 in 10^9 chance of failing to detect more errors," but this is length-bounded. Per BIP-173 and Samuel Dobson's "(Some of) the math behind Bech32 addresses," the block length n was chosen as 1023 = 32²−1 with minimum distance d=4, but "the exact generator polynomial was then picked to actually guarantee error detection of 4 errors in up to 89 characters"; bech32 strings are capped at 90 characters total. BIP-173 itself notes the checksum "is not always robust against the insertion and deletion of fewer than 5 consecutive characters." Beyond ~89–90 characters there is NO 4-error detection guarantee. Therefore for a hand-transcribed share code reaching ~118 characters, the guarantee does NOT hold — the report's concern is well-founded, and bech32 (^1) is the wrong choice at that length.
- (b) Discoverer/when: documented in the sipa/bech32 GitHub issue #51 and discussed on the bitcoin-dev mailing list in December 2019 (referenced from BIP-350). Precise individual attribution is UNVERIFIABLE from sources gathered this pass.

**PRIORITY 6 — CRC seam / KDF (partially verified)**
- **Quarkslab "VeraCrypt 1.18 Security Assessment"** (Bedrune & Videau, Ref. 16-08-215-REP, 17 Oct 2016, OSTIF-funded) — CONFIRMED verbatim from the report PDF: "CRC-32 is not a cryptographic hash function. Using it in this key derivation mechanism creates [a weakness] ... NCC Group's report recommends the use of HMAC with a cryptographic hash function. We concur. This recommendation should really be implemented. For the moment the problem has not been corrected, probably because it would break backward compatibility."
- **CRC-32 polynomial irreducibility — REFUTED, but the report's conclusion SURVIVES.** The standard CRC-32 polynomial (0x04C11DB7 / reversed 0xEDB88320, i.e. x^32+x^26+x^23+x^22+x^16+x^12+x^11+x^10+x^8+x^7+x^5+x^4+x^2+x+1) is NOT irreducible over GF(2); it factors (John D. Cook, "Polynomials used in CRC checksums": "it is far from irreducible"). By contrast the CRC-64-ISO polynomial x^64+x^4+x^3+x+1 IS irreducible mod 2. Therefore, if Stigge-Plötz-Müller-Redlich ("Reversing CRC – Theory and Practice," HU Berlin SAR-PR-2006-05, 2006) states the CRC-32 polynomial is irreducible and hence the state ring "is even a field ... where every polynomial ≠ 0 is invertible," that statement is FALSE for CRC-32 — a real source supporting a false claim. HOWEVER, the report's INJECTIVITY conclusion does not depend on irreducibility: the CRC state update is an affine map whose linear part is the companion matrix of a polynomial with nonzero constant term (constant term = 1), so the per-byte/per-bit state transition is a bijection regardless of whether the polynomial factors. The report's per-byte-invertibility conclusion therefore stands even though the cited justification (field structure) is wrong. **Action:** re-word the report to rest on invertibility of the CRC state transition, not on irreducibility/field structure.

**PRIORITY 11 — new developments that would change the plan**
- **Wide-block mode choice — decisive new datapoint.** NIST SP 800-197A (PRE-DRAFT Call for Comments, published 6 June 2025; comments due 6 Aug 2025) launched development of "cryptographic accordions" and proposes to "develop three general-purpose accordions: Acc128 to support typical usage (birthday bounds) with the Advanced Encryption Standard (AES); Acc256 ... **Recommendations**
1. **Immediately (Priority 1):** change the recovery-share encoder off BIP-173 bech32 (^1). For ~118-character MAC-bearing codes, bech32's 4-error/89-char guarantee is void. Adopt bech32m (constant 0x2bc830a3) at minimum, or preferably BIP-93 codex32, which is purpose-built for longer secret-sharing codewords. Threshold that would change this: if share codes can be guaranteed ≤89 characters, bech32's substitution guarantee is restored — but MAC-bearing codes cannot be, so plan for the long-code encoder.
2. **Correct the reports' factual errors before publication:** (a) R01's XCB confirmation date → 2 September 2024; (b) R25's "UC San Diego" → UC Davis / Regents of the University of California for US 2004/0131182; (c) add a note that ePrint 2024/1554 is a single evolving paper (GEM original → CRYPTO 2025 two-query version) so the two citations are not contradictory; (d) re-word the CRC-32 injectivity argument to rest on invertibility of the CRC state transition, NOT on polynomial irreducibility/field structure, since the CRC-32 polynomial is reducible.
3. **Keep HCTR2 as the wide-block mode:** it is unbroken, deployed in-kernel, patent-clear per its authors, and now the front-runner for NIST's Acc128 accordion. Re-evaluate only if NIST finalizes a different accordion or a beyond-birthday mode (HCTR+/GEM) reaches standardization with a public reference implementation.
4. **Treat AEZ as a don't-build** (R01 is right on the merits). If R25 keeps AEZ in scope, restrict its rationale to "patent-free" and drop any "viable" security framing.
